Для реализации данного технического задания (ТЗ) с учетом требований по изоляции слоев, наиболее подходит Clean Architecture (или Onion Architecture).

1. TodoApp.Domain (Core). Центральный слой. Не имеет зависимостей от других уровней или внешних библиотек.
2. TodoApp.Application (Logic). Слой бизнес-логики. Зависит только от Domain.
3. TodoApp.Infrastructure (Data). Реализация деталей хранения. Зависит от Domain (для реализации интерфейсов) и Application.
4. TodoApp.API (Transport). Точка входа. Зависит от Application и Infrastructure (только для регистрации зависимостей в DI).

Начнем с Domain Layer (Ядро)
Этот слой не должен иметь зависимостей от других уровней или внешних фреймворков.

Создаем solution с названием TodoApp. 
Создаем проект (как библиотеку) TodoApp.Domain. 
В нем создаем каталог Entities где будем хранить все сущности. 

Определение Entities: Создание сущности TodoList с необходимыми атрибутами и коллекции связей с "доверенными пользователями". Создаем класс TodoList для описания списка задач. Все свойства имеют private set. Это гарантирует, что объект нельзя перевести в невалидное состояние (например, сменить владельца или обнулить ID) извне. Необходимые атрибуты:
	Guid Id: 		Уникальный идентификатор списка.
	string Title: 	Название списка.
	Guid OwnerId: 	Идентификатор пользователя, который создал список (владелец).
		Эти три сущности придут в конструктор извне. Тут же есть валидация Title от 1 до 255 символов.
	DateTime CreatedAt: 	Время создания 
		Для простоты заполняю сразу при создании.
	List<Guid> SharedUserIds: 	Коллекция ID пользователей, которым открыт доступ. Для реализации требования ТЗ о том, что пользователь может делиться списком задач.
		1) Коллекцию делаем приватной и создаем методы для добавления и удаления пользователей, для чтения текущего состояния.  
		2) Так же нашел интересную возможность для списка именно под нашу задачу. Поскольку он приватный, то как то же его надо "отдавать" наружу. Я буду использовать AsReadOnly.
			Когда ты вызываешь _sharedUserIds.AsReadOnly(), .NET не копирует все данные в новый список. Вместо этого он создает обертку (ReadOnlyCollection), которая имеет ссылку на оригинальный _sharedUserIds, но без методов .Add(), .Remove(), .Clear(). 
		3) Владельца сюда не добавляю. Так получится возможным отличить владельца от того, кому дан доступ (мне кажется так более правильно). Так же проверяю, что бы владельца было нельзя добавить в этот список по ошибке.
		
Там же создаем класс User для идентификации участников системы и вносим в него все необходимые атрибуты:
	Guid Id: 			Уникальный идентификатор пользователя.
	string UserName: 	Имя пользователя.
		Оба параметра так же будут приходить в конструктор снаружи.

Обрати внимание, я в списке задач не создаю именно "список задач". В ТЗ я этого не вижу, и действия, описанные в ТЗ никак не манипулируют самим списком задач. Т.е. тут скорее про названия, пользователей и связи. Потому я решил упростить решение и оставить только Title.

Прежде чем создавать сервис, нам нужно определить контракт для работы с данными. В Clean Architecture это делается через интерфейсы в слое Domain. Это будет слой-посредник между бизнес-логикой (Domain) и источником данных (база данных, файлы, внешние API).

Создаем в каталоге Repositories интерфейс для описания методов ITodoListRepository

Для выполнения требований ТЗ создаем 5 методов
* Создать новый список задач.			AddAsync
* Изменить существующий список задач.	UpdateAsync
	посколько в MongoDB сущность хранится как бинарный JSON, то UpdateAsync будет технически выполнять замену документа целиком. Я выбрал этот вариант потому, что если бы мы обновляли только отдельные поля (например, только Title), мне бы пришлось писать сложную логику отслеживания изменений.
* Удалить существующий список задач.	DeleteAsync
* Получить один существующий список задач.	GetByIdAsync
* Получить список списков задач. 		GetPagedAsync
Следующие действия будут выполняться через загрузку и сохранение (обновление) сущности
* Добавить связь одного списка задач с указанным пользователем.
* Получить связи одного списка задач с пользователями.
* Удалить связь одного списка задач с указанным пользователем.

Этим мы достигаем решения следующих вопросов:
1) Использование Aggregate Root (Корень агрегации).
	Сущность TodoList является корнем агрегации. Это означает, что она сама полностью контролирует свое состояние и внутренние данные (включая список связей SharedUserIds).
2) Инкапсуляция бизнес-логики.
	Все правила реализованы внутри методов самой сущности в слое Domain. Репозиторию не нужно знать об этих правилах; его задача — просто сохранить объект целиком.
		Например, запрет на добавление владельца в список связей или валидация длины заголовка
3) Атомарность операций в NoSQL.
	Так как в качестве хранилища выбрана MongoDB, один список задач со всеми его связями хранится как один документ. Метод UpdateAsync заменяет или обновляет этот документ целиком, что гарантирует целостность данных без использования сложных транзакций.

Хотя в ТЗ говорится, что в запросах достаточно вернуть только идентификатор и название списка, на этом уровне я оставил возврат всей сущности. Мне кажется это более правильно, так как это уровень связи с базой данных и из базе я могу вернсть всю сущность, а проекцию я могу сделать на уровне Application. 
Если завтра нам понадобится отобразить в списке не только название, но и дату создания (CreatedAt) или количество участников, нам не придется менять интерфейс ITodoListRepository.

Application Layer (Logic)
Слой бизнес-логики, который координирует работу сущностей и репозиториев. Этот слой не знает ничего о MongoDB. Он знает только об интерфейсах из Domain и о том, какие данные хочет получить пользователь.
Service — это «оркестратор». Он не умеет сам писать в базу и не должен содержать в себе правила валидации (они в сущностях). Его работа — выстроить цепочку действий для выполнения конкретного бизнес-сценария.

Заметь, что на данном этапе реализована прикладная логика (Application Layer) без привязки к конкретному хранилищу. TodoListService взаимодействует исключительно с контрактами, определенными в ITodoListRepository. Физическая реализация методов (работа с MongoDB Driver, формирование фильтров и маппинг BSON) будет вынесена в слой Infrastructure, что обеспечивает полную независимость бизнес-логики от деталей реализации БД.
	Чистая демонстрация принципа Dependency Inversion (D в SOLID): высокоуровневые модули (Service) не должны зависеть от низкоуровневых модулей (Infrastructure), оба должны зависеть от абстракций (Interfaces).
Почему это важно зафиксировать:
	Тестируемость: Благодаря такому подходу можно написать Unit-тесты для TodoListService, подставив вместо реальной базы данных «заглушку» (Mock), которая будет имитировать поведение репозитория.
	Чистота кода: В сервисе нет ни одной строки, специфичной для MongoDB.

TodoApp.Infrastructure (Data)
Оффтоп про базу данных

Выбранная база не реляционная и в MongoDB данные хранятся в формате BSON (Binary JSON).	
	Collection (Коллекция) — аналог таблицы.
	Document (Документ) — аналог строки. Это объект, который может содержать вложенные массивы и другие объекты.
	Схема: MongoDB является schemaless (без схемы). Можно сохранить документ с любым набором полей, однако в Clean Architecture структура данных диктуется C#-классом TodoList.
	
В реляционных базах данные разделены по разным таблицам (TodoList и SharedUsers) и соединяютя через JOIN. В MongoDB предпочтительно хранить связанные данные внутри одного документа, если они всегда используются вместе.
	Сущность TodoList выступает как Aggregate Root (корень агрегации). Список SharedUserIds хранится прямо внутри документа списка задач. Это позволяет получить все данные за один запрос без тяжелых объединений таблиц.

В MongoDB драйвере для реализации обновления данных используется метод ReplaceOneAsync (аналог UpdateAsync из EF Core). При этом есть важный момент: этот метод заменяет весь документ целиком по его Id. Это гарантирует атомарность: либо сохранится весь объект со всеми изменениями в заголовке и связях, либо ничего.

Так же вопрос с идентификаторами. 
	В SQL часто используется автоинкремент (1, 2, 3...) или Guid.
	MongoDB по умолчанию использует свой тип ObjectId, но наш проект настроен на использование Guid. Драйвер MongoDB умеет автоматически преобразовывать Guid из C# в бинарный формат в базе.

Связи. 
	В ТЗ реализована связь «один список — много пользователей» через хранение массива Guid.
	В SQL мы бы создали промежуточную таблицу UserTodoLists. В MongoDB мы просто храним массив идентификаторов SharedUserIds внутри документа TodoList. При поиске используется фильтр AnyEq, который проверяет наличие конкретного Guid в этом массиве.

Кратко для сравнения
	SQL / EF Core			MongoDB
	Table (Таблица)			Collection (Коллекция)
	Row (Строка)			Document (Документ)
	Column (Колонка)		Field (Поле)
	Foreign Key / JOIN		Embedding (Вложение) или Reference (Ссылка)
	Where()					FilterDefinition / Find()

Так как в проекте используется Clean Architecture, вся эта специфика будет скрыта внутри TodoListRepository, и слой бизнес-логики (Application) не заметит разницы между SQLite и MongoDB.

Итак, на текущий момент готовы ядро (Domain) и бизнес-логика (Application), которые не зависят от конкретных технологий. Теперь необходимо реализовать «детали»: подключение к базе данных и выполнение операций, описанных в интерфейсе ITodoListRepository.

Создем новый проект TodoApp.Infrastructure типа Class Library. Он будет содержать физическую реализацию доступа к данным.
	Зависимости: Проект должен ссылаться на TodoApp.Domain (для реализации интерфейса репозитория) и TodoApp.Application.
	Библиотеки: Устанавливаем драйвер для работы с MongoDB (например, пакет MongoDB.Driver).
	
Далее создем класс MongoDbContext. Он выступает в роли фабрики подключений и инкапсулирует логику инициализации MongoClient и выбор базы данных. Предоставляет типизированный доступ к коллекции TodoLists (аналог DbSet в EF Core).
Обрати внимение, что тут можно было захардкодить строку подключения и имя базы. Но я намеренно сделал рзделение ответственности: параметры подключения (connectionString и databaseName) НЕ хранятся в коде инфраструктуры. Они будут передаваться через конструктор извне. Конкретные значения будут определены в слое TodoApp.API (в файле appsettings.json), который является точкой входа приложения.

По моей задумке это позволяет менять базу данных (например, на тестовую) без большой модификации кода проекта.
Если завтра надо будет перейти с MongoDB на SQLite, то нам надо сделать следующее:
	Создать новый проект TodoApp.Infrastructure.Sqlite.
	Написать класс SqliteTodoListRepository, который также имплементирует интерфейс ITodoListRepository.
	В проекте API (в Program.cs) заменить строчку регистрации с MongoDB на SQLite.
	Для остальной части системы ничего не изменится. Сервис TodoListService как просил ITodoListRepository, так и продолжает его просить. Ему все равно, какой драйвер «под капотом».

Ну или вторая выгода это тестирование.
	Например:
		Local: подключаемся к mongodb://localhost:27017 (база для разработки).
		Test: подключаемся к временной базе для прогона тестов.
		Production: подключаемся куда то в облако с авторизацией и SSL.
	Благодаря тому, что connectionString и databaseName не «захардкожены» в коде, а передаются в конструктор, можно переключаться между этими серверами просто меняя значения в файле конфигурации, не пересобирая (не компилируя) проект заново.

Далее нам нужно реализовать сам механизм работы с данными — класс TodoListRepository, который имплементирует интерфейс ITodoListRepository. Этот класс будет содержать всю специфическую логику MongoDB Driver, превращая команды Application Layer в запросы к базе.
Тут стоит обратить внимание на ТЗ. Нам нужно обеспечить:
	Атомарность: Полное обновление документа.
	Безопасность: Фильтрация данных по владельцу или правам доступа.
	Производительность: Пагинация и сортировка на уровне базы данных.

Пояснение по созданному классу TodoListRepository:
Конструктор
	Внедрение контекста через конструктор:
		Репозиторий получает MongoDbContext через конструктор (Dependency Injection).
		Инициализируется приватное поле _collection, которое получает прямую ссылку на коллекцию "TodoLists" в MongoDB. Это позволяет выполнять запросы к конкретной "коллеции" (аналогия: таблица), минуя необходимость повторного подключения в каждом методе.
			Это не копия данных: все списки задач из базы в этот момент не скачиваются. Создается объект-манипулятор (IMongoCollection), через который потом будем отправлять команды (добавить, удалить, найти). 
	Использование типизированного доступа:
		Использование IMongoCollection<TodoList> гарантирует, что драйвер будет автоматически мапить документы BSON в доменные сущности TodoList при выполнении операций чтения и записи.
		Если бы нам были нужны, например, пользователи, то был бы репозиторий для работы с ними, типизированный сущностью User.
		Замечание про автоматический маппинг.
			Несмотря на то, что драйвер MongoDB умеет автоматически преобразовывать базовые типы данных (такие как Guid или string), он не имеет доступа к приватным членам класса по умолчанию. Так как наша сущность TodoList спроектирована с защитой логики (приватные сеттеры и приватные поля), нам нужно будет явно "подсказать" драйверу через конфигурацию мапинга, как именно он должен записывать данные в эти закрытые поля и использовать наш конструктор. Это чуть чуть позже сделаю.
			
Работа с данными
	Добавление 
		Имплементируем AddAsync из интерфейса ITodoListRepository. Используется метод драйвера MongoDB под названием InsertOneAsync, который сохраняет объект как новый документ в коллекции.
	Обновление
		Используем ReplaceOneAsync, так как в рамках нашей архитектуры мы обновляем документ целиком.
		Различие с SQL: В реляционной базе ты бы делал UPDATE table SET column = value WHERE Id = .... В MongoDB ReplaceOneAsync делает «замену по образцу».
			Аргументы: Первый аргумент — это делегат (фильтр), который ищет запись в базе (аналог WHERE в SQL). Второй аргумент — это новое состояние объекта (данные), которое полностью запишется поверх старого.
			Отсутствие Change Tracking: в отличие от EF Core, драйвер MongoDB не «следит» за полями объекта в памяти. Поэтому мы явно передаем фильтр поиска и актуальный объект для перезаписи.	
	
		Требование атомарности операций из ТЗ.
			В MongoDB операция изменения одного документа всегда атомарна по умолчанию. Это означает, что когда мы вызываем ReplaceOneAsync, база данных гарантирует: либо весь документ будет полностью заменен на новый, либо (в случае сбоя) останется старая версия документа. Состояния «наполовину обновленного» списка задач существовать не может.
			В реляционных базах (типа SQLite) для достижения такой же атомарности при обновлении нескольких таблиц тебе пришлось бы использовать транзакции (BeginTransaction). В MongoDB при работе с одним документом транзакции не нужны — атомарность заложена в саму операцию замены документа.

	Удаление
		Используем DeleteOneAsync. Здесь также первым аргументом идет фильтр поиска по Id, чтобы база знала, какой именно документ уничтожить.
	Поиск по Id
		Цепочка вызовов: Find -> FirstOrDefaultAsync
			В MongoDB Driver выполнение запроса разделено на два этапа:
			.Find(l => l.Id == id): На этом этапе драйвер только формирует запрос (аналог IQueryable в EF Core). Он еще не идет в базу данных. Лямбда-выражение внутри говорит базе: «подготовься искать документ с таким-то идентификатором».
			.FirstOrDefaultAsync(): Это команда на немедленное выполнение. Драйвер отправляет запрос, берет первый найденный документ из результата и закрывает соединение. Если ничего не найдено, метод вернет null.
		Возвращаемый тип TodoList? (Nullable)
			Тут Nullable крайне важен для бизнес-логики. Он явно говорит слою Application, что:
				Списка с таким ID может не существовать.
				Вызывающий код (ваш TodoListService) обязан проверить результат на null, прежде чем обращаться к свойствам объекта, чтобы избежать NullReferenceException.
		Отсутствие фильтрации по пользователю
			Заметь, что в GetByIdAsync мы ищем строго по Id, не проверяя, кто запрашивает список (владелец или нет). Почему это правильно здесь? Репозиторий — это просто низкоуровневый инструмент доступа к данным.
			Где же будет проверка? Проверка прав доступа по правилам Clean Architecture должна находиться в слое Application (TodoListService), после того как объект будет получен из репозитория. Так мы и сделали в 
	Получение списка с пагинацией
		Метод GetPagedAsync является самым сложным в репозитории, так как он объединяет в себе три задачи: фильтрацию по правам доступа, сортировку и постраничный вывод (пагинацию).
		Для создания фильтра мы используем Builders<TodoList> — это специальный служебный класс из библиотеки MongoDB Driver, который предоставляет строго типизированный способ построения запросов к базе данных. Если в SQL мы писали запрос строкой (например, "WHERE OwnerId = ..."), то в MongoDB используем Builders, чтобы сконструировать этот запрос с помощью C#-кода.
			Фильтрация: 
				Реализовано требование ТЗ по ограничению видимости данных. 
					Используется фильтр Builders<TodoList>.Filter.Or, который выбирает документы, где 
						ИЛИ текущий пользователь является владельцем (Filter.Eq(l => l.OwnerId, userId)) 
						ИЛИ включен в список доверенных лиц (Filter.AnyEq(l => l.SharedUserIds, userId)). 
				Это гарантирует, что база данных вернет только разрешенный контент. Вместо того чтобы выкачивать все данные и фильтровать их в коде C#, мы заставляем базу данных делать это до передачи данных на сторону шарпа. 
			Сортировка: 
				Применен метод .SortByDescending(). Это обеспечивает выполнение требования ТЗ по отображению списков в порядке "от новых к старым".
			Пагинация: 
				.Skip((pageNumber - 1) * pageSize) — пропускает документы предыдущих страниц. 
				.Limit(pageSize) — ограничивает размер выборки текущей страницей.
			Эффективность: 
				Все операции (фильтрация, сортировка и пагинация) выполняются на стороне сервера MongoDB. В приложение передается уже готовый отфильтрованный и обрезанный набор данных.
			Заметь, что методы Find, SortByDescending и пр похожи на LINQ. Но это методы Fluent Interface самого драйвера MongoDB.Driver. Их намеренно сделали похожими на LINQ. Запрос сначала собирается в цепочку и потом хавершающий вызов .ToListAsync() асинхронно выполняет его и материализует результат в коллекцию объектов.

Регистрация маппинга сущностей (настройка BSON-маппинга)

Чтобы Infrastructure слой заработал с Domain слоем, нам нужно настроить BsonClassMap. Драйвер MongoDB по умолчанию не видит приватные поля и не знает, как использовать конструктор с параметрами. Мы создадим специальный конфигурационный класс MongoDbMapping, который выполнит "регистрацию" сущностей.

Итак по строкам
BsonClassMap 
	Механизм десериализации (преобразования BSON-данных из БД в объекты C#) и сериализации (обратного процесса)
	Это статический класс, входящий в состав библиотеки MongoDB.Driver. Он является частью подсистемы сериализации. Это своего рода «менеджер настроек», который управляет тем, как сериализатор (BsonSerializer) будет разбирать и собирать объекты.
	
cm.AutoMap()
	Стандартное действие "по умолчанию". Это заставляет драйвер просканировать класс и сопоставить все публичные свойства с геттерами и сеттерами.
	У нас, например, он найдет Title (у которого есть публичный геттер), но не сможет его записать при чтении из базы из-за private set. Однако AutoMap() полезен как базовая настройка, чтобы не прописывать новое публичное поле вручную, если оно появится в будущем и будет иметь публичный сеттер.
	
cm.MapIdField(c => c.Id);
	В MongoDB поле идентификатора в документе всегда называется _id.
	Хотя свойство Id публичное, драйверу нужно подтверждение, что именно это свойство в C# соответствует системному полю _id в BSON.
	Без этой строки: Драйвер может попытаться создать в базе два поля: _id (сгенерированный самой базой) и Id (наше свойство). Явный мапинг связывает их воедино.
	
cm.MapField("_sharedUserIds").SetElementName("SharedUserIds");
	MapField("_sharedUserIds")
		C# запрещает менять IReadOnlyCollection, а MapField позволяет драйверу наполнить коллекцию «в обход» интерфейса. 
			Мы даем драйверу доступ к private полю через Reflection. Свойство SharedUserIds в C# доступно только для чтения (IReadOnlyCollection). Драйвер MongoDB не может записать в него данные из базы стандартными средствами. MapField дает драйверу «черный ход» напрямую к приватному списку через Reflection.
	.SetElementName("SharedUserIds") 
		Так же по умолчанию драйвер ищет в базе поле с тем же именем, что и в коде (_sharedUserIds). Мы указываем, что приватное поле _sharedUserIds соответствует ключу SharedUserIds в BSON-документе и в базе данных оно будет храниться красиво: "SharedUserIds": [...].
	
cm.MapConstructor();
	Драйвер MongoDB при выполнении запросов (Find, GetByIdAsync) получает из базы набор байтов (BSON). Чтобы превратить их в объект TodoList в памяти C#, ему нужно вызвать какой-то конструктор.
	По умолчанию драйвер ищет конструктор без параметров (new TodoList()), создает пустой объект, а потом через сеттеры заполняет поля. Настройка конструктора MapConstructor указывает драйверу использовать другой (не по умолчанию, а конкретный) конструктор с параметрами для создания объектов при десериализации из базы данных. Фактически через MapConstructor мы говорим: «Чтобы воссоздать объект на основе данных из БД, возьми данные из поля Id в базе и засунь их в первый параметр конструктора, данные из Title — во второй» и так далее.
Вот на этом этапе мне кажется, что я наше у себя неточность.
	 Если использовать текущий конструктор до каждый раз дата создания будет перетираться CreatedAt = DateTime.UtcNow.
Решение: 
	Добавить и прописать приватный конструктор, который позволяет драйверу передать (восстановить) ту дату, которая уже лежит в базе. Я делаю его приватным, чтобы никто из программистов не мог случайно создать объект с «левой» датой — это разрешено только базе данных.
	Таким образом при создании будет использован публичный конструктор и дата будет записана автоматически, а при ВОСсоздании будет использован специальный, который не перетирает дату. Вот этот специальный и надо регистрировать. 
	Поскльку это данные из базы а не от пользователя, то валидация не нужна. Она уже была выполнена при записи в базу.
Флаги при поиске конструктора
	По умолчанию метод GetConstructors() возвращает только публичные конструкторы. Поскольку мы сделали конструктор для базы данных приватным, нам нужно явно приказать системе «искать глубже».
	BindingFlags.Instance
		Этот флаг говорит: «Ищи конструкторы, которые относятся к конкретному экземпляру объекта».
		Конструкторы всегда являются экземплярными методами (они создают экземпляр). Если этот флаг не указать, поиск может вернуть пустой результат, так как по правилам рефлексии нужно уточнить, ищем мы статические члены или экземплярные.
	BindingFlags.NonPublic
		Это ключевой флаг для нашего случая. Он говорит: «Ищи методы, уровень доступа которых не является public». Сюда входят private, protected и internal. Без этого флага рефлексия просто «проигнорирует» новый конструктор, считая его скрытым от внешнего мира.	
	Почему я использую [0]?
		GetConstructors возвращает массив. У нас сейчас:
			Один публичный конструктор (с 3 параметрами).
			Один приватный конструктор (с 5 параметрами).	
		Вызов GetConstructors с флагами вернет массив, содержащий только мой новый приватный конструктор. Соответственно, индекс [0] — это способ его достать.

При "поднятии" данных из базы можно встретить термин гидратация (Hydration) — это технический процесс «оживления» объекта в оперативной памяти приложения на основе данных, полученных из постоянного хранилища (базы данных). 
В контексте твоего проекта это превращение сырого BSON-документа из MongoDB в полноценный экземпляр класса TodoList в C#.
	Когда ты сохраняешь объект, происходит дегидратация (или сериализация): объект «высушивается» до состояния набора байтов или текста. Гидратация — это обратный процесс.
Основные этапы гидратации в моем коде:
	Извлечение данных: 
		Драйвер MongoDB считывает документ из коллекции.
	Выбор стратегии создания: 
		Поскольку нет публичного пустого конструктора, драйвер ищет альтернативный путь.
	Вызов конструктора (Constructor Injection): 
		Используя настройки MapConstructor, драйвер передает считанные из базы Id, Title, OwnerId, CreatedAt и SharedUserIds в параметры приватного конструктора.
	Заполнение состояния: 
		Внутри приватного конструктора данные присваиваются полям и свойствам объекта.
	Восстановление связей: 
		Приватное поле _sharedUserIds наполняется списком идентификаторов напрямую через Reflection, минуя публичный интерфейс IReadOnlyCollection.

	MongoDB Driver не ORM в привычном понимании, а драйвер. Он «общается» с базой через BSON (бинарный JSON). Регистрируем его как Синглтон. 
	Я прочитал, что есть специфика драйвера MongoDB: Внутри MongoDbContext создается MongoClient и он управляет пулом соединений (Connection Pool) с базой данных. Согласно официальной документации он должен быть синглтоном. Это экономит накладные расходы на сеть и память. К тому же драйвер MongoDB полностью потокобезопасен. Один экземпляр контекста может обслуживать сотни параллельных запросов от разных пользователей одновременно без риска перемешивания данных.

А вот что касается репозитория то он будет создаваться на каждый HTTP запрос. Scoped означает, что объект живет ровно столько, сколько обрабатывается один HTTP-запрос. Когда приходит запрос от Пользователя А, создается свой экземпляр репозитория. Когда приходит запрос от Пользователя Б, создается другой экземпляр.
	В простых репозиториях это может не вызвать проблем, но если добавить в репозиторий какое-то внутреннее состояние (например, кеширование данных пользователя), то данные Пользователя А могли бы «утечь» к Пользователю Б. Scoped гарантирует, что каждый запрос начинается с «чистого листа».
	
	Так же есть один момент. TodoListRepository зависит от MongoDbContext. Сейчас контекст — Singleton. В такой связке репозиторий мог бы быть синглтоном. Но! Если завтра ты сменить MongoDB на SQL (Entity Framework), то AppDbContext будет Scoped. Если репозиторий останется Singleton, а база станет Scoped, приложение упадет с ошибкой: "Cannot consume scoped service from singleton".
	Использование Scoped для репозиториев — это золотой стандарт, который делает архитектуру гибкой для смены базы данных.
	
	Ну и читал такое, что контекст — это тяжелый "мост" к базе (хранит пул соединений). Его накладно пересоздавать, поэтому его стараются делать один на всех. Репозиторий — это легкий "инструмент", который берет этот мост и выполняет конкретную задачу для конкретного юзера. Инструменты лучше выдавать новые на каждый запрос.

Как это работает в момент запроса?
Когда в API приходит запрос:
	.NET создает Scope (область видимости) для этого запроса.
	DI-контейнер создает TodoListRepository.
	DI-контейнер создает TodoListService и «вставляет» в него созданный репозиторий.
	DI-контейнер создает TodoListsController и «вставляет» в него сервис.
	Выполняется метод контроллера.
	Когда ответ отправлен пользователю, Scope закрывается, и сервис с репозиторием уничтожаются, освобождая память.

    * launchBrowser: false: Указывает, нужно ли автоматически открывать браузер при запуске. Для API часто ставят false, так как мы тестируем запросы через Swagger или Postman вручную.
* Сетевые настройки:
    * applicationUrl: Это адреса и порты, которые приложение зарезервирует на ПК. В профиле http это http://localhost:5098, в профиле https — защищенный адрес.
* Переменные окружения (environmentVariables):
    * ASPNETCORE_ENVIRONMENT: Development: Это критически важная строка. Она говорит приложению, что оно работает в режиме «Разработки». Именно благодаря этой строке в Program.cs срабатывает условие if (app.Environment.IsDevelopment()), которое включает Swagger (OpenAPI).

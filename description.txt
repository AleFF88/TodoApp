Для реализации данного технического задания (ТЗ) с учетом требований по изоляции слоев, наиболее подходит Clean Architecture (или Onion Architecture).

1. TodoApp.Domain (Core). Центральный слой. Не имеет зависимостей от других уровней или внешних библиотек.
2. TodoApp.Application (Logic). Слой бизнес-логики. Зависит только от Domain.
3. TodoApp.Infrastructure (Data). Реализация деталей хранения. Зависит от Domain (для реализации интерфейсов) и Application.
4. TodoApp.API (Transport). Точка входа. Зависит от Application и Infrastructure (только для регистрации зависимостей в DI).

Начнем с Domain Layer (Ядро)
Этот слой не должен иметь зависимостей от других уровней или внешних фреймворков.

Создаем solution с названием TodoApp. 
Создаем проект (как библиотеку) TodoApp.Domain. 
В нем создаем каталог Entities где будем хранить все сущности. 

Определение Entities: Создание сущности TodoList с необходимыми атрибутами и коллекции связей с "доверенными пользователями". Создаем класс TodoList для описания списка задач. Все свойства имеют private set. Это гарантирует, что объект нельзя перевести в невалидное состояние (например, сменить владельца или обнулить ID) извне. Необходимые атрибуты:
	Guid Id: 		Уникальный идентификатор списка.
	string Title: 	Название списка.
	Guid OwnerId: 	Идентификатор пользователя, который создал список (владелец).
		Эти три сущности придут в конструктор извне. Тут же есть валидация Title от 1 до 255 символов.
	DateTime CreatedAt: 	Время создания 
		Для простоты заполняю сразу при создании.
	List<Guid> SharedUserIds: 	Коллекция ID пользователей, которым открыт доступ. Для реализации требования ТЗ о том, что пользователь может делиться списком задач.
		1) Коллекцию делаем приватной и создаем методы для добавления и удаления пользователей, для чтения текущего состояния.  
		2) Так же нашел интересную возможность для списка именно под нашу задачу. Поскольку он приватный, то как то же его надо "отдавать" наружу. Я буду использовать AsReadOnly.
			Когда ты вызываешь _sharedUserIds.AsReadOnly(), .NET не копирует все данные в новый список. Вместо этого он создает обертку (ReadOnlyCollection), которая имеет ссылку на оригинальный _sharedUserIds, но без методов .Add(), .Remove(), .Clear(). 
		3) Владельца сюда не добавляю. Так получится возможным отличить владельца от того, кому дан доступ (мне кажется так более правильно). Так же проверяю, что бы владельца было нельзя добавить в этот список по ошибке.
		
Там же создаем класс User для идентификации участников системы и вносим в него все необходимые атрибуты:
	Guid Id: 			Уникальный идентификатор пользователя.
	string UserName: 	Имя пользователя.
		Оба параметра так же будут приходить в конструктор снаружи.

Обрати внимание, я в списке задач не создаю именно "список задач". В ТЗ я этого не вижу, и действия, описанные в ТЗ никак не манипулируют самим списком задач. Т.е. тут скорее про названия, пользователей и связи. Потому я решил упростить решение и оставить только Title.

Прежде чем создавать сервис, нам нужно определить контракт для работы с данными. В Clean Architecture это делается через интерфейсы в слое Domain. Это будет слой-посредник между бизнес-логикой (Domain) и источником данных (база данных, файлы, внешние API).

Создаем в каталоге Repositories интерфейс для описания методов ITodoListRepository

Для выполнения требований ТЗ создаем 5 методов
* Создать новый список задач.			AddAsync
* Изменить существующий список задач.	UpdateAsync
	посколько в MongoDB сущность хранится как бинарный JSON, то UpdateAsync будет технически выполнять замену документа целиком. Я выбрал этот вариант потому, что если бы мы обновляли только отдельные поля (например, только Title), мне бы пришлось писать сложную логику отслеживания изменений.
* Удалить существующий список задач.	DeleteAsync
* Получить один существующий список задач.	GetByIdAsync
* Получить список списков задач. 		GetPagedAsync
Следующие действия будут выполняться через загрузку и сохранение (обновление) сущности
* Добавить связь одного списка задач с указанным пользователем.
* Получить связи одного списка задач с пользователями.
* Удалить связь одного списка задач с указанным пользователем.

Этим мы достигаем решения следующих вопросов:
1) Использование Aggregate Root (Корень агрегации).
	Сущность TodoList является корнем агрегации. Это означает, что она сама полностью контролирует свое состояние и внутренние данные (включая список связей SharedUserIds).
2) Инкапсуляция бизнес-логики.
	Все правила реализованы внутри методов самой сущности в слое Domain. Репозиторию не нужно знать об этих правилах; его задача — просто сохранить объект целиком.
		Например, запрет на добавление владельца в список связей или валидация длины заголовка
3) Атомарность операций в NoSQL.
	Так как в качестве хранилища выбрана MongoDB, один список задач со всеми его связями хранится как один документ. Метод UpdateAsync заменяет или обновляет этот документ целиком, что гарантирует целостность данных без использования сложных транзакций.

Хотя в ТЗ говорится, что в запросах достаточно вернуть только идентификатор и название списка, на этом уровне я оставил возврат всей сущности. Мне кажется это более правильно, так как это уровень связи с базой данных и из базе я могу вернсть всю сущность, а проекцию я могу сделать на уровне Application. 
Если завтра нам понадобится отобразить в списке не только название, но и дату создания (CreatedAt) или количество участников, нам не придется менять интерфейс ITodoListRepository.

Application Layer (Logic)
Слой бизнес-логики, который координирует работу сущностей и репозиториев.

Создаем проект TodoApp.Application
Этот слой не знает ничего о MongoDB. Он знает только об интерфейсах из Domain и о том, какие данные хочет получить пользователь.

В ТЗ сказано: при получении списка достаточно ID и Названия. Создадим для этого легкую модель только с этими полями. TodoApp.Application/DTOs/TodoListBriefDto.cs
